01 · Alex Kim · 2026-03-31
Claude Code 源码泄露：虚假工具、挫败感正则、隐身模式，以及更多
来源：alex000kim.com · HN 722 points · 阅读原文 ↗

Anthropic 在 Claude Code 的 npm 包里意外附带了 .map 文件，完整可读的 TypeScript 源码就这么暴露了。这是一周内 Anthropic 的第二次意外泄露——上次是 model spec。Alex Kim 花了一个上午读完 HN 评论和泄露源码，按"辣度"排了个序。

Anti-distillation：用虚假工具污染竞争对手

源码里有个 flag 叫 ANTI_DISTILLATION_CC。开启后，Claude Code 会在 API 请求里附带 anti_distillation: ['fake_tools']，告诉服务端往 system prompt 里悄悄注入虚假的工具定义。

逻辑很直接：如果有人在录制 Claude Code 的 API 流量来训练竞品模型，这些假工具会污染训练数据。这个功能通过 GrowthBook feature flag 控制，只对第一方 CLI session 生效。

还有第二层机制——connector-text summarization。服务端会把 assistant 在 tool call 之间的文本做摘要加密签名。录 API 流量的人只能拿到摘要，看不到完整推理链。

但说实话，绕过并不难。MITM proxy 把请求里的 anti_distillation 字段去掉就行了，因为注入是服务端 opt-in 的。设一个环境变量 CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS=1 也能关掉。认真做 distillation 的人大概一个小时就能找到所有绕过方式。真正的保护可能还是靠法律，不是技术。

Undercover Mode：让 AI 假装自己不是 AI

undercover.ts 大概 90 行代码，做的事情是：在非 Anthropic 内部 repo 里使用 Claude Code 时，自动抹掉所有内部痕迹——"Capybara"、"Tengu"之类的内部代号，Slack 频道名，甚至"Claude Code"本身。

代码里写着一行注释："There is NO force-OFF. This guards against model codename leaks." 你可以强制开启，但没办法强制关闭。这是个单向门。

这意味着 Anthropic 员工在开源项目里提交的 AI 生成代码，不会有任何 AI 标注。隐藏内部代号没问题，但让 AI 主动伪装成人类——这是另一回事。

用正则表达式检测用户挫败感（没错，正则）

源码里有个正则表达式，用来检测用户是不是在发飙：

/\b(wtf|wth|ffs|omfg|shit(ty|tiest)?|dumbass|horrible|awful|piss(ed|ing)? off|piece of (shit|crap|junk)|what the (fuck|hell)|fucking? (broken|useless|terrible|awful|horrible)|fuck you|screw (this|you)|so frustrating|this sucks|damn it)\b/

一家 LLM 公司用正则做情感分析——挺讽刺的。但话说回来，用正则比调一次 LLM 推理来判断用户是不是在骂你，确实又快又便宜。

Native Client Attestation：JS 层之下的 DRM

API 请求里有个 cch=00000 占位符。在请求离开进程之前，Bun 的 Zig 原生 HTTP 层会把这五个零替换成一个计算出来的哈希。服务端用这个哈希验证请求确实来自真正的 Claude Code binary，不是伪造的。

占位符长度一样，所以替换不会改 Content-Length，也不需要重新分配 buffer。计算发生在 JS runtime 之下，JS 层完全看不到。说白了——这是 API 调用的 DRM，在 HTTP 传输层实现。

这也解释了 OpenCode 法律纠纷的技术背景。Anthropic 不只是口头要求第三方工具别用他们的 API——binary 本身会密码学证明"我是真的 Claude Code 客户端"。

KAIROS：还没发布的自主 Agent 模式

代码里到处都有一个叫 KAIROS 的 feature-gated 模式。从代码路径看，这是个未发布的自主 agent 模式，包括：

  • /dream 技能——"nightly memory distillation"（夜间记忆蒸馏）
  • 每日只追加的日志
  • GitHub webhook 订阅
  • 后台 daemon worker
  • 每 5 分钟的 cron 定时刷新

这大概是整个泄露里最大的产品路线图发现。一个常驻后台、持续运行的 agent 的基础架构已经搭好了。

我的看法：有人觉得无所谓，因为 Google 的 Gemini CLI 和 OpenAI 的 Codex 已经开源了。但那些公司开源的是 agent SDK（工具包），不是旗舰产品的完整内部布线。真正的损失不是代码——代码可以重构。而是 feature flag 背后的产品路线图，这些战略意外一旦泄露就收不回来了。

02 · Anthropic GitHub Skills · 2026-03-22 ~ 03-25
Claude API Skill 持续自动同步——官方 Skill 规范在悄悄进化
来源：github.com/anthropics/skills · 查看提交记录 ↗

3 月 22 日到 25 日之间，Anthropic 的官方 Skills 仓库连续出现了多个自动同步提交，全部标记为 chore: update claude-api skill [auto-sync]。

发生了什么

这些提交不是人工写的——是 Anthropic 内部的自动化管道把 Claude API 文档的变更同步到 Skill 定义文件。短短 4 天内出现 5 次 auto-sync，说明 Claude API 本身在这段时间经历了密集的迭代。

为什么值得关注

Anthropic 的 Skills 仓库是 Claude Code 理解"我能做什么"的知识来源。每一次 SKILL.md 的更新都直接影响 Claude Code 在使用 Anthropic SDK 时的行为——新的参数、新的用法、弃用的旧接口。

这种自动同步机制本身就值得学习：官方文档变更 → 触发 CI → 自动更新 Skill 定义 → Claude Code 下次读取时就拿到最新信息。不需要人工干预，不会遗漏。

对 Skill 工程的启示

如果你在构建自己的 agent 系统，Anthropic 的做法给了一个清晰的模式：

  • Skill 是活的文档，不是写完就放着的静态文件。需要有机制让它跟上游保持同步。
  • Auto-sync 的粒度控制很重要——每个 Skill 独立更新，不是一把全推。
  • 提交信息的标准化（[auto-sync] tag）让人和机器都能快速区分手工更新和自动同步。
实际影响：如果你在用 Claude Code 的 claude-api Skill，现在拿到的版本和两周前已经不一样了。建议定期 pull 最新的 Skill 文件，或者直接对着官方仓库做 diff，看看哪些 API 用法有变化。

03 · Simon Willison · 2026-04-03
当 AI 开始自己找漏洞：安全研究的认知冲击
来源：simonwillison.net · 引用 Thomas Ptacek 原文 · 阅读原文 ↗

Simon Willison 在 4 月 3 日集中分享了几条关于 AI 对安全研究领域冲击的内容。核心是 Thomas Ptacek 的一篇文章《Vulnerability Research Is Cooked》——漏洞研究这个领域，正在被 AI agent 彻底改变。

Agent 在漏洞研究上的能力，是结构性的

Ptacek 的核心判断很直接：在未来几个月内，coding agent 会从根本上改变漏洞挖掘的实践和经济学。这不是渐进式改善，而是台阶式跃迁。大量高影响力的漏洞研究，会变成"把 agent 指向一个代码仓库，然后输入'帮我找 zero day'"。

You can't design a better problem for an LLM agent than exploitation research.
——Thomas Ptacek

为什么 agent 在这件事上特别强？三个原因叠在一起：

  • 内置知识：在你喂它一个 token 之前，frontier LLM 已经编码了海量代码库之间的关联。Linux KVM hypervisor 和 hrtimer 子系统有没有关系？模型知道。
  • Bug class 库：stale pointer、integer mishandling、type confusion、allocator grooming——所有已知的 bug 类型和利用方式都在模型权重里。
  • 不会累：漏洞发现本质上是模式匹配 + 约束求解 + 可达性分析。结果是明确的成功/失败。Agent 不会觉得无聊，你让它搜到天荒地老它就搜到天荒地老。

Linux 内核安全报告的暴增：从每周 2-3 份到每天 5-10 份

Simon 同时引用了两位 Linux 内核维护者的证词。Greg Kroah-Hartman（Linux 内核维护者）说：

几个月前，我们收到的都是 "AI slop"——AI 生成的安全报告，一眼就能看出是错的。但一个月前，世界变了。现在我们收到的报告是真的，是好的，是用 AI 做的。

Willy Tarreau（HAProxy 核心开发者）给了具体数字：安全报告从两年前的每周 2-3 份，到去年的每周约 10 份（多出来的全是 AI slop），到今年初变成了每天 5-10 份——而且大部分是正确的。他们不得不引入更多维护者来处理。更离谱的是，现在每天都能看到重复报告：同一个 bug，被两个不同的人用不同的工具同时发现。

供应链攻击的社会工程学也在升级

同一天，Simon 还记录了 Axios npm 供应链攻击的事后分析。攻击者的手法高度定制化：伪装成某公司创始人，搭建一个看起来完全真实的 Slack workspace，安排 MS Teams 会议，然后以"你的系统需要更新"为由让维护者安装了 RAT（Remote Access Trojan）。

Simon 的感受很真实：我也经常在最后一刻才安装 Webex 或 Teams，时间紧迫时什么都点 yes。所有维护重要开源项目的人都需要了解这种攻击方式。

值得想想的事：AI 同时在攻防两端加速。Agent 能帮你找漏洞，也能帮攻击者生成更精准的社会工程攻击。Linux 内核安全报告从 slop 到正确报告的转变只用了一个月——这种速度对所有软件维护者来说都是一个需要认真面对的变化。

04 · Alex Kim · 2025-01-11
在 Kubernetes 上编排 LLM 微调：SkyPilot + MLflow 完整指南
来源：alex000kim.com · 阅读原文 ↗

大家讨论 LLM 通常关注模型层面的事——选什么基座、用什么数据、LoRA 还是全参数。但实际上，高效管理计算资源和实验追踪同样重要，甚至更容易被忽视。Alex Kim 这篇文章拿 SkyPilot 和 MLflow 这两个开源工具，演示了如何把 LLM 微调的基础设施搭得可复现、可扩展。

一个 YAML 启动整个训练

SkyPilot 的核心理念是把计算资源抽象成声明式配置。一个 sky.yaml 文件定义了资源需求（比如 H100:8）、环境安装、训练命令——然后 sky launch 一行搞定。不需要手动管 K8s pod，不需要写 Dockerfile，不需要处理 GPU 调度。

关键配置大概长这样：

resources:
  cloud: kubernetes
  accelerators: H100:8

setup: |
  pip install -U -r requirements.txt
  python generate_train_dataset.py

run: |
  accelerate launch --num_processes 8 train.py

不只是 K8s——SkyPilot 支持几乎所有云厂商（AWS、GCP、Azure），同一份 YAML 换个 cloud: 字段就能跑在不同平台上。

分布式训练的 MLflow 集成：进程管理是关键

多 GPU 分布式训练环境下，MLflow 的日志记录必须小心处理。多个进程同时往 MLflow 写数据会导致竞争条件、重复记录甚至数据损坏。Alex 的方案很实际：

  • 只在主进程上初始化 MLflow，通过 trainer.accelerator.is_main_process 判断
  • 训练前保存 run_id——因为 trainer.train() 完成后会自动关闭 MLflow run，不提前存 ID 后面就没法补日志
  • 每个节点设独立的 system metrics ID，GPU 利用率、显存占用这些指标才能对应到具体的机器

实用建议

  • 用 HF Accelerate 时，所有 MLflow 操作前必须检查 is_main_process
  • 用云厂商时开 managed jobs——spot 抢占或硬件故障时能自动恢复
  • 用 sky queue 排多组超参实验，用 sky logs 查历史日志
  • 训练完了 sky down dev 把集群关掉——别忘了，H100 按小时计费
为什么选这篇：不是最新的文章，但对于任何想认真做 LLM 微调的团队来说，基础设施这层的复杂度经常被低估。SkyPilot + MLflow 这个组合全开源、跨云、支持分布式——比自己从零搭 K8s GPU 调度省太多时间了。

Anthropic 推出 Usage Bundles，额外赠送使用额度庆祝上线
来源：Anthropic Official · Hacker News · 2026-04-03 · 原文 ↗

Anthropic 正式推出 Claude 使用包（Usage Bundles）——一种新的打包定价模式，让重度用户可以按套餐购买 API 使用额度，而非纯按 token 计费。为庆祝上线，Anthropic 为现有订阅用户额外赠送使用额度。

  • 新的定价模式直接影响 Claude 生态的经济学——对高频 API 调用场景（如 Agent 系统、批量处理）尤其有吸引力
  • 这是 Anthropic 在商业化上的又一步：从按量付费到套餐制，降低重度用户的边际成本
  • 值得关注后续是否会影响 Claude Code 订阅模式——目前已有 HN 用户反馈 Claude Code 订阅政策可能有变化
